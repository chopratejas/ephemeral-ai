## Technical Specification: Ephemeral.ai (JIT Agentic Infrastructure)

**Version:** 1.0

**Status:** Hackathon Concept / Open Source

**Infrastructure Provider:** DigitalOcean (Gradient, App Platform, Droplets, Spaces)

**Core Philosophy:** Infrastructure should only exist for the "lifetime of a thought."

---

## 1. Executive Summary

**Ephemeral.ai** is a "Just-In-Time" (JIT) infrastructure orchestrator. Traditional cloud setups keep servers running 24/7, incurring idle costs. Ephemeral.ai uses **DigitalOcean Gradient** to run a "Neural Gateway" that parses complex user tasks, predicts the exact hardware and software environment needed, provisions a temporary **DigitalOcean Droplet**, executes the task, and destroys the Droplet immediately upon completion.

It transforms "Cloud Computing" into "Cognitive Computing," where the LLM’s intent is the primary driver of resource allocation.

---

## 2. System Architecture

The system consists of four primary layers:

### A. The Neural Gateway (The Brain)

* **Host:** DigitalOcean Gradient (GPU-backed Inference).
* **Model:** Llama 3 or DeepSeek-Coder (fine-tuned/prompt-engineered for SysAdmin tasks).
* **Function:** Receives raw text prompts (e.g., *"Process this 500MB CSV and give me a summary"*). It outputs a **Manifest JSON** containing:
* **Resource Spec:** (e.g., `s-1vcpu-2gb`)
* **Environment:** (e.g., `python:3.10-slim` or `node:18`)
* **Bootstrap Script:** The actual code to execute the task.



### B. The Orchestrator (The Nervous System)

* **Host:** DigitalOcean App Platform (Serverless/Static).
* **Tech Stack:** Go or Python (FastAPI).
* **Function:** The bridge between the LLM and the DigitalOcean API. It listens for the Manifest JSON, calls the **DO Cloud API** to spin up a Droplet, and monitors the lifecycle.

### C. The Thought-Node (The Muscle)

* **Host:** DigitalOcean Droplet.
* **Life Expectancy:** 30 seconds to 10 minutes.
* **Function:** A clean, isolated environment that pulls the "Bootstrap Script" from the Orchestrator, executes it, sends the output back to the user, and signals its own "death" to the Orchestrator.

### D. The State-Vault (The Memory)

* **Host:** DigitalOcean Spaces (Object Storage).
* **Function:** Since Droplets are destroyed, any persistent data (logs, processed files) is synced to Spaces before the "Delete" command is issued.

---

## 3. The "Manifest" Protocol

The core innovation is the JSON structure generated by the Gradient LLM. It must be strictly typed to ensure the Orchestrator can execute it without human intervention.

```json
{
  "task_id": "uuid-12345",
  "infra_requirement": {
    "slug": "s-1vcpu-1gb",
    "region": "nyc3",
    "image": "ubuntu-22-04-x64"
  },
  "runtime_env": {
    "language": "python",
    "dependencies": ["pandas", "numpy"]
  },
  "payload": "print(df.describe())",
  "termination_policy": "immediate_after_output"
}

```

## 4. Implementation Plan (Hackathon Sprints)

### Phase 1: The "Grand Master" Prompt (0-4 Hours)

Using **Claude Code**, build a robust system prompt for the model running on **DigitalOcean Gradient**. The goal is "Zero-Shot Infrastructure."

* *Test:* Can the LLM correctly identify that a video-transcoding task needs more RAM than a text-parsing task?

### Phase 2: The Fast-Provisioner (4-12 Hours)

Build the Go/Python worker that handles the **DigitalOcean API** calls.

* **Optimization:** Use "Droplet Tagging" to track all ephemeral nodes.
* **Claude Code Task:** *"Generate a high-concurrency Go worker that creates a Droplet with a Cloud-Init script to auto-execute a Python payload."*

### Phase 3: The State Sync (12-18 Hours)

Implement a "Final Breath" script. Before the Droplet shuts down, it must `rclone` or `s3cmd` its results to **DO Spaces**.

* **Safety:** Ensure the Orchestrator verifies the upload before calling the `DELETE /droplets/{id}` endpoint.

### Phase 4: The "Shadow" Dashboard (18-24 Hours)

A React/Tailwind frontend that visualizes:

* Real-time "Inference-to-Infra" latency.
* Money saved vs. an "Always-On" server.
* The live logs of the "Thought-Node" as it wakes up, works, and dies.

---

## 5. Security & Constraints

* **API Key Isolation:** The LLM *never* sees the DO API Keys. Only the Orchestrator (running on App Platform) has access to environment variables.
* **Sandboxing:** Thought-Nodes are isolated Droplets. Even if the LLM generates malicious code, it only destroys a temporary, empty VM.
* **Rate Limiting:** Hard-cap the number of concurrent Droplets to prevent "Recursive Inference" from blowing the hackathon budget.

## 6. Future Vision (Post-Hackathon)

* **Neural Load Balancing:** Multiple Gradient models predicting traffic spikes before they happen.
* **Cross-Provider Ephemerality:** Using the same logic to spin up resources on AWS/GCP based on price-arbitrage, managed by the DO "Brain."
